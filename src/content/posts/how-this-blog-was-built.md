我终于把这个博客从模板状态整理成了自己的样子。
这不是一个很复杂的项目，但它对我来说像是一个小小的落点：以后遇到的问题、做过的判断、学到的东西，不必都散在聊天记录、临时笔记和脑子里。

## 为什么想重新搭一个博客

其实一直都有"该有个博客"的想法，但每次都卡在同一个地方：想做的东西太多，最后什么都没做。

以前用过 GitHub Pages + Jekyll，也折腾过 Notion 公开页面、语雀之类的方案。但要么维护太重（每次想改点什么都得翻文档），要么平台锁死（内容不在自己手里，想搬走很麻烦），要么写着写着就变成"技术文档堆砌"——失去了写的动力。

这次想清楚了几件事：

第一，博客首先是写给自己看的。几年后回来看，能想起当时为什么这么做、为什么这么想，那就够了。第二，越简单越好。我不想每次发篇文章还要跑构建、调样式、修兼容性。第三，内容和平台解耦。哪天不想用这个框架了，Markdown 文件一搬就走。

所以这次的目标很明确：**一个能用、够轻、写得舒服的博客**。没有流量 KPI，没有装修焦虑，不给维护留债。

## 为什么选 Astro / Fuwari / Cloudflare Pages

这次选型其实没花太多时间纠结，核心诉求就那么几条。

**Astro** 之前就关注过。它最打动我的一点是"静态优先"——默认输出就是纯静态 HTML，不需要跑服务端渲染，也没有复杂的运行时。写博客这个场景，Astro 几乎就是天选：内容驱动、岛屿架构按需加载、Markdown/MDX 一等公民。而且它的 Island 架构很实用，偶尔想加点交互组件不用硬上 React/Vue 全家桶。

**Fuwari** 是一个基于 Astro 的博客模板。坦白说最开始选它就是因为好看——干净、暗色主题舒服、字体也顺眼。用下来发现它的集成做得挺全：Tailwind、Svelte 组件、Expressive Code 代码高亮、Pagefind 搜索、Swup 页面切换，基本开箱即用。省去了从零搭样式的功夫。

**Cloudflare Pages** 纯粹是图省事。Git 仓库一推，自动构建发布，免费额度对个人博客来说完全够用。自定义域名、HTTPS、全球 CDN 都是点几下的事。之前用过 Vercel，也很好，但域名、构建、HTTPS 放在一个地方管理，少一个服务商就少一点操心。

整体思路就是：**静态生成 + Git 驱动 + CDN 分发**，每层都尽量简单。

## 这次做了哪些初始化

从模板到能用的状态，做的事情其实不多，但也比想象中多一点点。

**第一步是改站点身份。** 换站点名、改描述，头像暂时用模板自带的没换。Fuwari 的配置集中在 `src/config.ts` 里，改起来很方便。顺便把语言设为中文。

**第二步是清理模板内容。** Fuwari 带了不少示例文章（Markdown 示例、视频嵌入、图片展示之类的），改成草稿状态保留下来，方便以后参考格式。也顺便熟悉了一下它的文章目录结构和 frontmatter 格式。

**第三步是确认构建流程。** 项目自带的 `pnpm build` 会把 Astro 构建和 Pagefind 搜索索引生成串在一起。我跑了几遍，确认构建通过、搜索可用。项目还自带了一个 `pnpm new-post` 命令，配置一下就能用，自动生成带 frontmatter 的新文章模板，以后写新东西不用手敲模板头了。

**第四步是准备接入自定义域名。** 接下来打算在 Cloudflare 那边配一下，现在是先在本地都准备好了，等域名解析到位就能上线。

## 遇到的小问题

虽然整体算顺利，但中间还是卡了几个地方。

**pnpm check 暴露的跨框架类型问题。** 这个项目混用了 Svelte 和 Astro 组件，类型检查的时候会报一些跨框架的类型不匹配——比如 Svelte 组件里引用的类型在 Astro 语境下解析不一致。搞了好一会儿才意识到不是代码写错了，是 Astro 的类型推断对这种混用场景覆盖没那么完整。后来把这类问题记进了功能文档，提醒自己以后遇到类似报错先排查框架边界。

**归档页的 category 类型没对齐。** frontmatter 里的 `category` 字段允许为空，但归档页组件的类型定义只接受字符串，导致 `pnpm check` 过不去。检查之后统一了一下类型定义，两边对上就好了。

**.codegraph 本地数据库不该提交。** 开发过程中 CodeGraph 生成了一个 `.codegraph/` 目录，里面是本地索引文件。有几次差点一起提交上去——这种本地生成的数据进 git 毫无意义，反而会给别人带来困扰。后来在 `.codegraph/.gitignore` 里补了忽略规则。

**示例文章改草稿后构建依然通过。** 有次我把所有示例文章设成草稿，想看构建会不会跳过，结果是静默通过，没有任何提示。不算 bug，但说明构建流程对草稿的处理方式和直觉不太一样——它只管过滤输出，不管告诉你"这里有 N 篇草稿被跳过了"。

## 之后想写什么

博客搭起来只是第一步，真正重要的是之后能不能持续写。

计划中的方向大概有这么几个：

**技术实践**会是最主要的板块。工作中遇到的问题、项目里的设计决策、新技术的学习笔记，适合沉淀下来的就整理成文章。不求面面俱到，但求自己以后回看能看懂。

**踩坑记录**是我个人很看重的一个类型。很多问题搜了半天找到一个 Stack Overflow 回答解决了，但下次遇到还得再搜一遍。写下来既是帮自己巩固记忆，运气好的话可能也能帮到别人。

**读书和学习笔记**。技术书也好、非技术书也好，读完不写点什么很快就忘了。整理笔记的过程本身就相当于二次学习。

**工具链和效率**相关。平时折腾开发工具、编辑器配置、自动化脚本这些小东西，零零碎碎但挺有意思，适合写成短文分享。

**生活观察**偶尔也会写一点。毕竟博客不只是技术堆砌，它首先是自己的空间。

希望能保持一个不算太高但稳定的频率。写比完美重要。

这就是这个博客搭起来的全部故事了。很简单，但总算开始了。
